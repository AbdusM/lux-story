#### 1. CORE PHILOSOPHY & VISION

**E-001: "Trojan Horse" Career Assessment Paradigm**
- **Source:** Lines 25-27
- **Specification:** Looks/feels like premium indie game, but invisibly tracks 50+ WEF 2030 skills
- **Philosophy:** "Career discovery through contemplation, not examination"
- **Dependencies:** Requires hidden assessment layer, pattern tracking, skill mapping
- **Thinking:** This isn't just a feature - it's the entire product strategy. Every game feature must serve this dual purpose.

**E-002: Design Pillars (10 Commandments)**
- **Source:** Lines 144-157, 764-786
- **10 Principles:**
  1. Feel Comes First (30-second engagement test)
  2. Respect Player Intelligence (discovery over explanation)
  3. Every Element Serves Multiple Purposes
  4. Accessible Depth (easy to learn, hard to master)
  5. Meaningful Choices (visible consequences, no right answers)
  6. Friction is Failure (confusion = design failure)
  7. Emotion Over Mechanics (feelings > actions)
  8. Show, Don't Tell (environmental storytelling)
  9. Juice is Not Optional (feedback for every action)
  10. Kill Your Darlings (remove what doesn't serve core)
- **Application:** Decision-making filters for all features
- **Thinking:** These commandments create constraints that enable creativity. Each one implies evaluation systems and quality checks.

**E-003: Red Flags to Avoid (6 Anti-Patterns)**
- **Source:** CLAUDE.md
- **Anti-Patterns:**
  1. The Iceberg Game (90% features hidden)
  2. Developer's Delight (features that excite devs, confuse players)
  3. Progressive Paralysis (hiding features to reduce overwhelm)
  4. Invisible Value Prop (uniqueness not clear)
  5. Tutorial Crutch (design by instruction not intuition)
  6. Feature Graveyard (systems nobody uses)
- **Purpose:** Quality guardrails during development
- **Thinking:** The negative space is as important as the positive. These define what NOT to build.

#### 4. DIALOGUE SYSTEM

**E-016: Graph-Based Navigation (DAG)**
- **Source:** Lines 220-256, 314-323
- **Structure:** Directed Acyclic Graph
- **Component:** DialogueGraphNavigator (lib/dialogue-graph.ts)
- **Function:** Graph traversal, condition evaluation, choice validation
- **Thinking:** DAG prevents cycles but also prevents returning to past nodes. What if we allowed cycles for recursive conversations? What about graph visualization for debugging?

**E-017: DialogueNode Data Structure**
- **Source:** Lines 363-378, 745-762
- **Fields:**
  - `id` (string) - Unique identifier
  - `speakerId` (CharacterId) - Which character speaking
  - `text` (string) - Dialogue content with rich text markup
  - `choices` (DialogueChoice[]) - Optional array of 2-4 player options
  - `conditions` (StateCondition[]) - Entry requirements
  - `stateChanges` (StateChange[]) - Effects on selection
  - `transitions` ({ next: string, condition?: StateCondition }[]) - Next nodes
- **Validation:** speakerId validated via isValidCharacterId()
- **Thinking:** Rich structure - each node is mini-state machine. What about voice recording metadata? Translation keys? Accessibility text?

**E-018: StateCondition System**
- **Source:** Lines 365, 319
- **Purpose:** Entry requirements for dialogue nodes
- **Types:** Trust thresholds, pattern levels, knowledge flags, completed arcs
- **Evaluation:** StateConditionEvaluator component
- **Example:** `{ type: 'trust', characterId: 'samuel', min: 5 }`
- **Thinking:** Boolean logic for branching. What about weighted conditions? Probability-based unlocks? Time-based conditions?

**E-019: StateChange System**
- **Source:** Lines 367, 376
- **Purpose:** Effects triggered by node entry or choice selection
- **Types:** Pattern increments, trust modifications, flag sets, arc completions
- **Application:** Immutable transformations via GameStateUtils.applyStateChange()
- **Example:** `{ type: 'pattern', pattern: 'analytical', delta: +2 }`
- **Thinking:** State mutations carefully controlled. What about compound state changes? Conditional state changes? Delayed state changes?

**E-020: Accept/Reject/Deflect Choice Pattern**
- **Source:** Lines 239, 118-129 (Addendum)
- **Framework:** Inkle's three-response taxonomy
- **Types:**
  - **Accept:** Cooperative stance ‚Üí Helping/Patience patterns
  - **Reject:** Confrontational stance ‚Üí Analytical/Building patterns
  - **Deflect:** Evasive/Chaotic stance ‚Üí Exploring patterns
- **Philosophy:** Avoids complex branching trees while maintaining agency
- **Thinking:** Elegant taxonomy. But what about silence as 4th option? What about mixed responses that combine two stances?

**E-021: "Two Reasonable People" Test**
- **Source:** Line 245
- **Rule:** Would two different players make different choices?
- **Purpose:** Validates that choices are meaningful, not fake
- **Failure:** If everyone chooses same option, it's a fake choice
- **Application:** Quality check during dialogue writing
- **Thinking:** Brilliant validation method. What about analytics dashboard showing choice distribution in real-time? Auto-flagging fake choices?

**E-022: Choice Count Constraint (2-4 Options)**

- **Range:** Minimum 2, maximum 4 options per node
- **Reasoning:** Fixed-height container (140px), prevent choice paralysis
- **Psychology:** 3 options optimal (Goldilocks), 4 allows nuance
- **Thinking:** UI constraint drives design. What if this varied by player preference? What about accessibility option for fewer choices?
